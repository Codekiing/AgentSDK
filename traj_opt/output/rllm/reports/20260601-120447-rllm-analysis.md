## 问题发现

### 1. PPO_KL系统的为零 — Policy未有效更新 [🔴严重] [影响: rllm-config + verl配置]

**现象**: 所有4个run的PPO_KL均为0.000000，无论模型大小(7B/0.5B)或lr(1e-7~5e-6)
**证据**: 
- run_1780281634 (7B, lr=1e-7): KL=0.000734 (仅0.0007, effectively 0)
- run_1780284483 (7B, lr=1e-6): KL=0.000000 (96步)
- run_1780285930 (7B, lr=5e-6): KL=0.0055 (仅此run有非零KL, mode collapsed)
- run_1780283875 (0.5B, lr=5e-6): KL=0.000000 (128步)
- Round 1 (0.5B, lr=5e-6): KL=0.000000 (48步) — 同样为0

**诊断**: KL=0有3种可能:
1. Reference model与policy model相同 (VERL中ref_path未正确配置)
2. KL loss_weight=0或use_kl_loss=false
3. Log-prob计算在两个模型上完全一致 (意味着模型权重相同)

**关键洞察**: Round 1的KL也为0但val-acc=0.488(stable)，Round 2的val-acc持续下降。KL=0本身不是致命问题(Round 1证明可以训练), 但**KL=0 + entropy崩塌 + 大num_problems** 的组合是致命的。

**置信度**: 高 (4个run + Round 1 cross-validation)

### 2. 验证背离 + 训练崩溃 [🔴严重] [影响: rllm-config]

**现象**: val-acc从step 10的0.444持续下降到step 128的0.334, 每10步必降
**证据**:
- Step 10: val-acc=0.444, Step 50: 0.372, Step 100: 0.359, Step 128: 0.334
- Monotonic decline — 无任何反弹
- Train score oscillating around 0.46 — 训练侧看不出问题

**诊断**: 这种"train side flat, val side declining"模式是过拟合的典型特征:
- 512 problems × 1 epoch → 模型见过每个样本仅1次
- 但等效batch_size=8×8×4=256 → 每个step的gradient noise来自problem difficulty variance而非policy improvement signal
- 模型在训练集上exploit了问题难度分布的模式(验证集上不适用)

**置信度**: 高 (val trend数据100%支持)

### 3. Entropy崩塌 [🔴严重] [影响: rllm-config]

**现象**: entropy 0.2885→0.0917, 从step 1开始线性下降, 在step ~110跌破0.1警戒线
**证据**: 
- 1st half entropy mean=0.123, 2nd half entropy mean=0.124 (看似稳定)
- 但趋势明确: 从0.29→0.09, 全程未反弹
- 对比Round 1: entropy 0.28→0.22, 48步仅下降0.06

**对比分析**:
- Round 1: entropy下降速度 = 0.06/48 = 0.0013/step
- Round 2: entropy下降速度 = 0.20/128 = 0.0016/step
- Round 2下降更快且持续时间更长, 等效entropy_loss = 0.197 vs 0.061

**诊断**: 不带entropy_coeff + num_problems=512 → 大量噪声驱动策略向确定性方向坍缩
**置信度**: 高

### 4. num_problems=512 严重过载 [🔴高] [影响: rllm-config]

**现象**: 512问题导致训练肤浅, 与Round 1形成鲜明对比
**证据**:
- Round 1: 32 problems × 2 epochs = 有效接触64次 | val-acc=0.488
- Round 2: 512 problems × 1 epoch = 有效接触512次但每个仅1次 | val-acc=0.334
- 模型需要在少量高质量数据上多轮精炼, 而非海量数据的浅层扫描
- 0.5B模型的容量决定了它需要repetition来巩固学习

**诊断**: 问题数增加10倍导致:
1. 梯度信号被问题难度差异淹没
2. 模型无法在单次接触中学习复杂推理模式
3. 每次step面对完全不同的问题 → 无持续学习

**置信度**: 高 (Round 1 vs Round 2实证对比)

### 5. Round 2 中7B模型实验的教训 [ℹ️信息] [影响: rllm-config]

**现象**: 7B模型在3个配置下均表现异常
**数据**:

## 建议的 Patch 内容

### Patch 1: rllm-config — 0.5B安全参数上限 [P0]

基于Round 1实证(32 problems, val-acc=0.488)和Round 2实证(512 problems, val-acc=0.334):
- num_problems: 安全上限64 (0.5B), 推荐32-48
- 理由: 0.5B模型需要重复接触少量数据进行深度精炼
- 多epoch训练(2-3)优于单epoch海量数据

### Patch 2: rllm-config — 默认配置修复 [P0]

从当前默认(batch=8, problems=512)回归验证有效的保守配置:
- lr: 5e-6 → 3e-6 (缓解entropy崩塌)
- num_problems: 512 → 48
- epochs: 严格保证2轮
- entropy_coeff: 显式启用 (0.001)
- temperature: 保持0.85
- batch_size: 8 → 4

### Patch 3: rllm-config — Entropy崩塌紧急检测 [P0]

在训练配置验证阶段新增:
- 如果entropy_coeff=0或未配置 → WARN: "entropy可能崩塌"
- 监控规则: entropy<0.15 → WARN, entropy<0.1 → 紧急停止建议

### Patch 4: rllm-train — 训练结束状态写入 [P0]

### Patch 5: rllm-config — 验证背离检测 [P1]

在训练配置的monitor阶段新增:
- 每10步检查val-acc趋势
- 连续3次val-acc下降 → ALERT: "验证背离, 建议停止训练检查配置"
- 当前Round 2数据: val-acc从0.444降到0.334 (11次连续下降, 全部被忽略)

### Patch 6: rllm-config — 7B模型参数指南 [P1]

基于Round 2中3次7B实验:

- 7B temperature: ≥0.5 (0.3过低, 熵会快速塌缩)

- 必须启用entropy_coeff (≥0.001)
